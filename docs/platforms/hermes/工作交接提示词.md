# 工作交接提示词

这份文档提供的是“可直接复制给下一位 agent”的中文提示词。

使用原则：

- 先选最接近当前任务的一条
- 不要把所有提示词叠在一起
- 如果是高风险动作，仍要保留人工确认

## 1. 通用接手提示词

适用：

- 新开一个 Codex / Claude / 其他 agent 会话
- 需要让对方快速接手这条 `Codex + Hermes + AgentBridge` 主链

```text
你现在接手的是 Windows 主力机 AI Agent 分层架构维护工作。

先不要猜，也不要直接改 live 环境。请按下面顺序建立上下文：

1. 先读 D:\CODE\local-ai-dev-orchestrator\README.md
2. 先读 D:\CODE\local-ai-dev-orchestrator\docs\Windows主力机-AI-Agent-分层架构-v1.6.md
3. 再读 D:\CODE\local-ai-dev-orchestrator\docs\当前交接摘要.md
4. 再读 D:\CODE\local-ai-dev-orchestrator\docs\接手检查单.md
5. 如涉及 live 运行，再读 D:\CODE\local-ai-dev-orchestrator\snapshots\agentbridge-20260628\docs\implementation-status.md

关键边界：
- 维护仓：D:\CODE\local-ai-dev-orchestrator
- live 运行面：C:\Users\sciman\Documents\AgentBridge
- 外部参考源码：D:\CODE\external\local-ai-dev-orchestrator-references
- 不要把真实 .env 或 private-local 下的 volume 备份加入 git
- 不要把 Hermes 产出的 task 当成对 Codex 的直接授权
- 如果当前在做 `Phase 0`，要先区分：`10000:10000 + probe-only bootstrap` 的 comparative 成功样本不等于 accepted 运行模型
- 当前已经可以区分为：`derived image + runtime_user 10001:10001 + container_start_user 0:0` 已在 live 通过正式验收，现已是 accepted truth
- 同时还要区分：`P0-1 cap_drop:[ALL]` 已在这条新 accepted runtime model 上复核，仍 blocked；`P0-3` 已收口为 `platform_na / 非 Phase 1 前提`

先输出：
1. 你理解的当前落点
2. 当前轮你准备做的最小闭环
3. 哪些是 repo-side，哪些是 live-side

然后再动手。
```

## 2. 只读核查提示词

适用：

- 我只想确认当前状态
- 我不想让 agent 直接改 live 环境或重跑 Hermes

```text
请只做只读核查，不要修改 live 环境，不要重启，不要拉起新服务。

先核对：
- D:\CODE\local-ai-dev-orchestrator 当前 git 状态
- C:\Users\sciman\Documents\AgentBridge 当前 docs / scripts / snapshot 真值
- D:\CODE\external\local-ai-dev-orchestrator-references 当前参考仓是否还在预期路径
- D:\CODE\local-ai-dev-orchestrator\references\updates\reference-refresh-latest.md 当前是否仍代表最近一次参考仓刷新结果

输出要求：
- 先给短结论
- 再分清：维护仓真值、live 运行真值、外部参考仓真值
- 明确哪些已验证、哪些未验证
- 不要把文档状态说成 live 已重跑
```

## 3. 单变量升级提示词

适用：

- 要升级 Hermes tag
- 要切换 provider 组合
- 要补一个新的受控放行能力

```text
请按“单变量变更”执行，不要并发改多个变量。

先做：
1. 指出这次唯一要改的变量
2. 记录当前 known-good 锚点
3. 说明回滚入口

执行顺序固定：
- contract
- gates
- boundary
- snapshot

结果里必须明确：
- 改了什么
- 没改什么
- live 是否真的重跑
- snapshot 是否更新
- 如果失败，停在哪一步，为什么
```

## 4. 文档同步提示词

适用：

- 当前工作只是整理方案、计划、清单、交接材料
- 不打算改运行脚本或重跑 Hermes

```text
这轮只做文档与索引同步，不做 live bring-up。

请以当前 repo truth 和已保存快照为准，更新：
- README
- 计划
- 任务清单
- 交接摘要
- 提示词 / 检查单

要求：
- 不要把“材料已补齐”说成“live 已再次验收”
- 明确区分 repo-side done 与 live-side done
- 提交前确认真实 .env 与 private-local 仍处于 ignored
```

## 5. 刷新参考仓提示词

适用：

- 想统一刷新 `D:\CODE\external\local-ai-dev-orchestrator-references` 下的上游参考仓
- 想把更新结果落成差异摘要

```text
请只处理外部参考仓刷新，不要顺手改 Hermes bring-up 主链。

先做：
1. 检查 D:\CODE\external\local-ai-dev-orchestrator-references 下各仓是否 clean
2. 先运行 D:\CODE\local-ai-dev-orchestrator\scripts\refresh-reference-repos.ps1 -FetchOnly -SkipDirtyRepos
3. 注意该脚本默认只刷新 `codex/plugins/modelcontextprotocol/servers`；只有本次任务真的涉及 `openai-agents-python`、`hermes-agent` 或 archive 仓时，才通过 `-RepoNames` 显式扩展
4. 再决定是否执行 pull --ff-only

输出要求：
- 明确哪些仓 behind、哪些已同步
- 给出最新摘要文件路径
- 不要把外部参考仓更新说成 live AgentBridge 已重跑
```

适用：

- 当前目标是复核 `verify-hermes-boundary.ps1` 当前证据
- 不想顺手扩大到整个 Hermes bring-up 体系

```text
当前已知边界：
- Windows 侧 v1.6 bring-up 已经跑通
- 当前 boundary 锚点是 D:\CODE\local-ai-dev-orchestrator\snapshots\agentbridge-20260628\docs\verify-hermes-boundary-20260628-225841-414.json
- 当前脚本应直接观测到 10001:10001 的 hermes chat 进程

本轮要求：
- 不要顺手重构整个架构
- 先核对 live 与 snapshot 的 verify-hermes-boundary.ps1 是否一致
- 再确认 service_uidgid_present 是否仍为 true
- 如出现回退，再给出最小修复路径和验证证据
```

## 7. 工作交接总结提示词

适用：

- 一轮工作结束，要把结果交给下一个人或下一个 agent

```text
请把本轮工作写成可交接摘要。

格式要求：
- 先给一句短结论
- 再分四段：Goal / Changes / Verification / Risks
- 必须写明：
  - 当前落点
  - 当前目标归宿
  - 哪些文件已更新
  - 哪些是 live 已验证
  - 哪些只是文档或快照层面的收口
  - 下一位接手建议先看哪 3 个文件

不要写成泛泛周报，要写成“下一位马上能继续”的交接材料。
```

- 不打算改运行时代码

```text
这轮只做 v2 文档同步，不改运行时代码，不重跑 Hermes，不改 live AgentBridge。

请完成：
1. 新建 v2 正式文档：

3. 清理旧口径：

输出要求：
- 先给短结论
- 再写清：哪些是新建文档，哪些是同步更新
- 不要把文档落盘说成 Wave 1 已实现
```

## 9. Wave 1 纯本地 MVP 提示词

适用：

- 当前要继续 `Wave 1 / Phase 1` 纯本地实现

```text
请只做 Wave 1 纯本地 SDK MVP。

固定边界：
- single worker
- single repo
- pure local tasks
- SDK first
- codex exec fallback
- workspace-write
- approval=never
- no auto_merge
- no push / PR
- no auto_continue

三类任务固定为：
1. 代码小修/重构
2. 文档生成/整理
3. 本地脚本执行 + 契约校验

注意：
- smoke 可与 Phase 0 并行
- 正式 accepted 须在 Phase 0 的边界收口已经明确后生效
- 不要引入 watcher / lease / VM / network allowlist / PAD
- 如果顺手看到 `P0-2` comparative success evidence，也不要把它写成 `Phase 0 accepted`
- 如果顺手看到 `P0-1/P0-3`，也不要再把它们写成“待判定”；当前固定口径应分别是 `still blocked` 与 `platform_na`

结果中必须区分：
- smoke 已做
- accepted 尚未达成 或 已达成

如果当前仓内已经存在 `runtime/host-orchestrator/`：
- 先核对 `W1-T01 ~ W1-T07` 是否已经 repo-side 完成
- 明确区分：
  - repo-side fake-first 闭环已完成
  - `P1-1 structured usage repo-side` 是否已落地
  - `P1-2 git 工作树` 是否已收口
  - live `Codex SDK` 真机执行是否已验收
- 不要把“pytest + contract test 通过”直接说成 `Phase 1 accepted`
```
