# Hermes 高自治终态重构：最终版

## 1. 定位

这份文档是本仓 `target truth` 的正式真源。

它定义的是：

- `v1.6` 之后的高自治演进终态
- `Wave 0 ~ 5d` 的正式波次
- `active truth` 与 `target truth` 的关系

它**不**替代当前 `active truth`。当前已验收真相仍以：

- [Windows主力机-AI-Agent-分层架构-v1.6.md](D:/CODE/hermes-agent/docs/Windows主力机-AI-Agent-分层架构-v1.6.md)

## 2. 总纲

终态仍是高自治四平面，但首版必须先做“官方能力优先的纯本地 Codex-Native MVP”，再逐步补自建 orchestrator、Windows VM GUI lane 和学习闭环。

两层真相并存：

- `active truth` = `v1.6` 可审计批处理闭环
- `target truth` = 本文定义的高自治四平面

## 3. 六条收敛结论

1. `Hermes` 不碰桌面；`Codex` 是唯一通用执行面；`Windows VM` 是正式 GUI lane；`AgentBridge` 继续是唯一证据总线。
2. 控制面固定为 `SDK first / codex exec fallback`；首版不自定义私有 RPC。
3. 常驻的是 `Host Orchestrator + heartbeat`；现有 `Hermes` 容器不升为常驻 supervisor。
4. 波次固定为：`Wave 0 = Phase 0 收口 -> Wave 1 = pure local SDK MVP -> 再叠加 watcher / lease / 路由 / 学习闭环 / GUI lane`。
5. 运行态进入 `SQLite control-plane.db`；`AgentBridge` 只承载任务定义、最终结果和证据。
6. `PAD` 不是首版执行面，只在 `Wave 4+` 且证明值得时再评估。

## 4. 四平面

### 4.1 Evidence Plane

- 继续只用 `AgentBridge`
- 固定目录：
  - `tasks/`
  - `results/`
  - `artifacts/`
  - `memory-promotions/`
  - `skills-drafts/`
  - `logs/`
- 固定硬规则：
  - task 不原地改写
  - 一个 task 一个正式 result
  - `Hermes` 文本是不可信输入
  - `skills-drafts` 不自动安装

### 4.2 Control Plane

- 主体：`Host Orchestrator`
- 语言：`Python 3.11+`
- 正式消费面：官方 [Codex SDK](https://developers.openai.com/codex/sdk)
- 定时唤醒：官方 `thread automation / heartbeat`
- 自建能力：
  - watcher
  - 原子认领
  - lease
  - 崩溃恢复
  - 路由分发
  - 失败协议
  - handoff 接管点

### 4.3 Learning Plane

- `Hermes` 容器只做按需学习作业
- 不作为常驻平台
- Windows 学习面固定为：
  - `AgentBridge` 证据
  - `Codex sessions`
  - `memories/skills`
  - `Hermes` 按需学习作业
- 不依赖 `Chronicle` 作为 Windows 基线

### 4.4 Execution Plane

- `host_local`
  - 本机 `Codex SDK`
  - `workspace-write`
  - `approval=never` 仅限低风险
  - `network=off`
- `remote_non_gui`
  - 目标主机本地 `remote runner`
  - 主机仅通过 `SSH` 触发/回传
  - Windows 上不把官方 `Remote connections` 当正式跨机控制面
- `vm_gui`
  - Windows VM 内本地 `vm-runner`
  - 读取 host 分发的任务包
  - 在 VM 内本地调用 `Codex SDK / Computer Use`
  - 结果回写 `AgentBridge`

## 5. Hermes / Orchestrator 身份

- `调度脑` = 新写 `host-orchestrator`
- `学习脑` = 按需启动的 `Hermes` 容器
- 禁止把现有 `Hermes` 容器升级成常驻平台

### Wave 1

  - `single worker`
  - `single repo`
  - `pure local tasks`
  - `SDK first`
  - `codex exec fallback`
  - `workspace-write`
  - `approval=never`

## 7. 默认安全模型

- 主机 worker：
  - `sandbox=workspace-write`
  - `approval=never` 仅限低风险
  - `network=off`
  - 独立 `CODEX_HOME`
  - 独立 `worktree`
- 风险三档：
  - 低风险自动完成
  - 中风险停可接管点
  - 高风险中断等待接管
- 主机不做常驻无人值守 GUI
- 任意 GUI 自动化进 `Windows VM`
- 自进化永不自动改写：
  - `AGENTS.md`
  - `CLAUDE.md`
  - `.claude/settings`
  - sandbox / approval / rules / security config
  - worker profile
  - 联网白名单
  - 主流程代码
- `FastAPI / OTel` 仅 `Wave 2+`，且：
  - `127.0.0.1`
  - 本机鉴权
  - 文件导出

## 8. 关键风险

- `app-server` 协议可能变化：
  - 消费面锁 `SDK`
  - 不直接耦合 `app-server`
- `heartbeat` 不是编排器：
  - 只做定时唤醒
  - watcher/lease/恢复全自建
- `network_proxy` 本机未证实：
  - `Wave 1-3` 默认纯本地
- `PAD unattended` 有 license / RDP / 锁屏 / 登出代价：
  - `Wave 4+` 才评估
- `Win -> Win` 不能直接靠官方 remote 控制：
  - GUI 走 VM 内本地执行

## 9. 验收

1. worker 控制面只用官方 `Python SDK`
2. `Wave 1` 三类任务与 `20 / 3 / 0 / 0 / 0 / 0` 验收口径一致
3. `AgentBridge` 无运行状态字段
4. `Hermes` 容器仍按需
5. `vm-runner + AgentBridge 文件总线` 是唯一 GUI lane 正式口径
6. 所有升级继续遵守 `contract -> gates -> boundary -> snapshot`
