# LAR-P0A-010 Q0 primary-source research

Date: 2026-07-16
Scope: contract research only; no live probe, runtime/config/state mutation, approval or implementation.
Sources: Microsoft Learn / Windows API and uv official documentation only.

## 1. Successor blocker: pre-resume child environment observation

### What the platform documents

- `CREATE_SUSPENDED` creates the primary thread in a suspended state and that thread does not run until `ResumeThread` is called. Therefore code inside that child cannot call `GetEnvironmentStringsW`, serialize its environment or report a digest before `ResumeThread`. [Process Creation Flags: `CREATE_SUSPENDED`](https://learn.microsoft.com/windows/win32/procthread/process-creation-flags#flags)
- `CreateProcessW` copies the caller-supplied environment block into the new process during creation. A Unicode block requires `CREATE_UNICODE_ENVIRONMENT`; its last string terminator plus block terminator occupy four zero bytes, or two UTF-16 NUL code units. [CreateProcessW: `lpEnvironment`](https://learn.microsoft.com/windows/win32/api/processthreadsapi/nf-processthreadsapi-createprocessw#parameters)
- `GetEnvironmentStringsW` returns only the environment of the *calling* process. It exposes no API for a parent to read an arbitrary suspended child's environment block. [GetEnvironmentStringsW](https://learn.microsoft.com/windows/win32/api/processenv/nf-processenv-getenvironmentstringsw)

### Consequence

The current acceptance phrase **"verifies the child-observed block before resume" is not supported by the documented Windows API model**. Before `ResumeThread`, the parent can prove only the canonical input buffer, its digest/length, the flags and the successful process-creation result. Actual child self-observation is necessarily post-resume. Treating the two as simultaneous would turn a fixture assertion into fictitious platform evidence.

### Supported replacement protocol

1. **Pre-resume parent proof:** construct the canonical UTF-16 block; verify key/value grammar, case-insensitive uniqueness, sort order and exact terminal NULs; persist the block digest, code-unit count, policy identity and `CreateProcessW` flags in the launch intent; create with `CREATE_UNICODE_ENVIRONMENT | CREATE_SUSPENDED | EXTENDED_STARTUPINFO_PRESENT`. This is a parent-input proof, not child observation. [Changing Environment Variables](https://learn.microsoft.com/windows/win32/procthread/changing-environment-variables) [CreateProcessW](https://learn.microsoft.com/windows/win32/api/processthreadsapi/nf-processthreadsapi-createprocessw)
2. **Post-resume Q0 child proof:** use a dedicated no-write probe whose first application action calls `GetEnvironmentStringsW`, canonicalizes the returned block and reports its digest through the qualified stdout handle. Compare that digest with the persisted pre-resume digest; mismatch is `platform_incompatible` and must fail qualification. [GetEnvironmentStringsW](https://learn.microsoft.com/windows/win32/api/processenv/nf-processenv-getenvironmentstringsw) [Pipe Handle Inheritance](https://learn.microsoft.com/windows/win32/ipc/pipe-handle-inheritance)
3. **Production meaning:** a successful generation-bound Q0 qualifies the parent serializer/CreateProcess composition. Per ordinary worker launch, the hard pre-resume gate remains parent-buffer verification; it cannot truthfully be called per-child read-back. Requiring every real target to self-verify before any target code runs would need a separately specified trusted bootstrap/trampoline and another process/effect protocol; Windows does not provide an `exec`-style replacement here.

This is a semantic change to the frozen v3.24 acceptance, not editorial clarification. A successor should replace `child-observed ... before resume` with distinct `pre_resume_parent_environment_proof` and `post_resume_q0_child_environment_observation` requirements.

## 2. Process creation, handles, Jobs and EOF

- A `STARTUPINFOEXW` attribute list is supplied to process creation with `EXTENDED_STARTUPINFO_PRESENT`. The list can contain multiple key/value attributes. [InitializeProcThreadAttributeList](https://learn.microsoft.com/windows/win32/api/processthreadsapi/nf-processthreadsapi-initializeprocthreadattributelist#remarks) [STARTUPINFOEXW](https://learn.microsoft.com/windows/win32/api/winbase/ns-winbase-startupinfoexw)
- `PROC_THREAD_ATTRIBUTE_JOB_LIST` is a list of Job handles assigned to the child in the specified order and is documented for Windows 10 / Windows Server 2016 and newer. This supports job assignment as part of `CreateProcess`, avoiding a documented post-create `AssignProcessToJobObject` window. [UpdateProcThreadAttribute: `JOB_LIST`](https://learn.microsoft.com/windows/win32/api/processthreadsapi/nf-processthreadsapi-updateprocthreadattribute#parameters)
- `PROC_THREAD_ATTRIBUTE_HANDLE_LIST` identifies handles to inherit. Every listed handle must itself be inheritable, pseudo handles are forbidden, and `bInheritHandles` must be `TRUE`. [UpdateProcThreadAttribute: `HANDLE_LIST`](https://learn.microsoft.com/windows/win32/api/processthreadsapi/nf-processthreadsapi-updateprocthreadattribute#parameters)
- With `STARTF_USESTDHANDLES`, `hStdInput`, `hStdOutput` and `hStdError` must be inheritable and `bInheritHandles=TRUE`; process creation does not validate that the supplied values are semantically correct. Exact object type, direction, access and role equality therefore remain contract/Q0 obligations. [STARTUPINFOW: `STARTF_USESTDHANDLES`](https://learn.microsoft.com/windows/win32/api/processthreadsapi/ns-processthreadsapi-startupinfow#members) [CreateProcessW remarks](https://learn.microsoft.com/windows/win32/api/processthreadsapi/nf-processthreadsapi-createprocessw#remarks)
- Windows permits inherited handles to refer to the same kernel objects with the same access. The project's non-empty exact stdio list, distinct-role rule and forbidden-handle catalog are intentionally narrower product policy, not extra guarantees supplied by `CreateProcess`. [Inheritance: inheriting handles](https://learn.microsoft.com/windows/win32/procthread/inheritance#inheriting-handles)
- Child processes normally remain in their parent's Job unless breakaway flags are enabled. `BREAKAWAY_OK` and `SILENT_BREAKAWAY_OK` are opt-ins; "no breakaway" is therefore represented by their absence and must be read back, not by a separate no-breakaway flag. [Job Objects: managing processes](https://learn.microsoft.com/windows/win32/procthread/job-objects#managing-processes-in-jobs) [JOBOBJECT_BASIC_LIMIT_INFORMATION](https://learn.microsoft.com/windows/win32/api/winnt/ns-winnt-jobobject_basic_limit_information#members)
- `KILL_ON_JOB_CLOSE` terminates associated processes only when the **last** Job handle closes. A retained/duplicated handle defeats that trigger. `TerminateJobObject` terminates all processes currently associated with the Job; neither API by itself proves that no external Job handle or pipe writer exists. [Job Objects: managing Job objects](https://learn.microsoft.com/windows/win32/procthread/job-objects#managing-job-objects) [TerminateJobObject](https://learn.microsoft.com/windows/win32/api/jobapi2/nf-jobapi2-terminatejobobject)
- `JOBOBJECT_BASIC_PROCESS_ID_LIST` reports assigned PIDs and may require a larger buffer when `NumberOfProcessIdsInList < NumberOfAssignedProcesses`; for nested Jobs it includes child Jobs. It is a state query, not PID-incarnation proof, so the contract still needs PID plus creation-time identity. [JOBOBJECT_BASIC_PROCESS_ID_LIST](https://learn.microsoft.com/windows/win32/api/winnt/ns-winnt-jobobject_basic_process_id_list)
- Anonymous-pipe EOF is conditional on **all** write handles being closed. Parent copies must close promptly; process termination closes that process's handles; a leaked writer prevents EOF. `ReadFile` may report the closed anonymous-pipe condition as `FALSE/ERROR_BROKEN_PIPE`, so a verifier must accept documented EOF forms rather than require only a zero-byte successful read. [Pipe Handle Inheritance](https://learn.microsoft.com/windows/win32/ipc/pipe-handle-inheritance) [ReadFile remarks](https://learn.microsoft.com/windows/win32/api/fileapi/nf-fileapi-readfile#remarks) [Anonymous Pipe Operations](https://learn.microsoft.com/windows/win32/ipc/anonymous-pipe-operations)

Windows documents the prerequisites for exit/kill-to-EOF, but not a universal bounded EOF deadline after controller crash or response loss. That bound must be a controller policy backed by exact-handle Q0 and fail-closed timeout evidence.

## 3. Windows environment block and comparison

- Environment strings have `name=value` form, must be sorted alphabetically by name using a case-insensitive Unicode order independent of locale, and the block is double-NUL terminated. Normal environment-variable names cannot contain `=`. [Changing Environment Variables](https://learn.microsoft.com/windows/win32/procthread/changing-environment-variables) [Environment Variables](https://learn.microsoft.com/windows/win32/procthread/environment-variables)
- `CREATE_UNICODE_ENVIRONMENT` distinguishes Unicode from ANSI input. On modern Windows, Microsoft documents no technical total environment-block limit, while one user-defined variable is limited to 32,767 characters; any smaller aggregate ceiling is a Local AI Runtime resource policy and must not be presented as an OS limit. [Process Creation Flags](https://learn.microsoft.com/windows/win32/procthread/process-creation-flags#flags) [Environment Variables: limits](https://learn.microsoft.com/windows/win32/procthread/environment-variables)
- `CompareStringOrdinal(..., bIgnoreCase=TRUE)` provides a non-linguistic, locale-independent Windows comparison primitive. Restricting catalog keys to ASCII makes a deterministic ASCII uppercase/lowercase fold sufficient for uniqueness; a general language/runtime `casefold()` is not established by these Windows docs as equivalent. [CompareStringOrdinal](https://learn.microsoft.com/windows/win32/api/stringapiset/nf-stringapiset-comparestringordinal)
- Microsoft Learn's normal environment grammar says names cannot contain `=`; the reviewed pages do not normatively specify drive-current-directory pseudo entries such as `=C:`. Rejecting every leading-`=` entry and never inheriting the ambient block is a valid narrower policy, but `=C:` behavior must not be claimed as a documented ordinary-variable requirement. [Changing Environment Variables](https://learn.microsoft.com/windows/win32/procthread/changing-environment-variables) [GetEnvironmentStringsW](https://learn.microsoft.com/windows/win32/api/processenv/nf-processenv-getenvironmentstringsw)

## 4. DPAPI and purpose separation

- By default, `CryptProtectData` is typically decryptable only with matching user logon credentials and usually on the same computer; roaming profiles are an exception. `CRYPTPROTECT_LOCAL_MACHINE` broadens decryption to any user on that computer and is unsuitable for a current-user-only envelope. [CryptProtectData remarks](https://learn.microsoft.com/windows/win32/api/dpapi/nf-dpapi-cryptprotectdata#remarks) [CryptProtectData flags](https://learn.microsoft.com/windows/win32/api/dpapi/nf-dpapi-cryptprotectdata#parameters)
- `pOptionalEntropy` participates in protection and the same bytes are required for unprotect. `szDataDescr` is a readable description stored with the ciphertext; it is not documented as a cryptographic purpose tag. [CryptProtectData parameters](https://learn.microsoft.com/windows/win32/api/dpapi/nf-dpapi-cryptprotectdata#parameters) [CryptUnprotectData parameters](https://learn.microsoft.com/windows/win32/api/dpapi/nf-dpapi-cryptunprotectdata#parameters)
- DPAPI adds a MAC, but Microsoft warns that corruption can yield different errors and in some cases may succeed with corrupted output; robust applications should add their own integrity checks. [CryptUnprotectData remarks](https://learn.microsoft.com/windows/win32/api/dpapi/nf-dpapi-cryptunprotectdata#remarks)

DPAPI alone does not prove `QuarantineKeyEnvelope` versus `RuntimeIntegrityKeyEnvelope` purpose separation, wrong-purpose rejection, suspended-only restore, generation binding or single consumption. Those require separate random content keys, distinct versioned entropy/purpose bytes, authenticated application metadata/integrity, ACL isolation and DB state-machine tests. The official API also does not promise literal SID-byte identity; use "current-user credential scope" plus Full-Q0 behavior, not an overstated DPAPI guarantee.

## 5. Disk reserve, accounting and quota capability

- `GetDiskFreeSpaceExW.lpFreeBytesAvailableToCaller` is quota-aware and may be less than total free bytes. It is an observation, not an atomic reservation; concurrent allocation can invalidate a prior reading. [GetDiskFreeSpaceExW](https://learn.microsoft.com/windows/win32/api/fileapi/nf-fileapi-getdiskfreespaceexw)
- `FILE_ALLOCATION_INFO` sets allocation size, independently from EOF except that EOF cannot exceed allocation. The filesystem protocol states unused allocation beyond EOF is freed on last-handle close. A persistent reserve therefore needs allocation and EOF aligned to the target size, followed by close/reopen read-back; `SetEndOfFile` alone sets file size and does not itself prove allocation size. [FILE_ALLOCATION_INFO](https://learn.microsoft.com/windows/win32/api/winbase/ns-winbase-file_allocation_info) [FileAllocationInformation protocol](https://learn.microsoft.com/openspecs/windows_protocols/ms-fscc/0201c69b-50db-412d-bab3-dd97aeede13b) [SetEndOfFile remarks](https://learn.microsoft.com/windows/win32/api/fileapi/nf-fileapi-setendoffile#remarks)
- Sparse files can expose large logical zero ranges without allocating equivalent disk space; compressed files can also use less physical space. A "fully allocated" reserve must at minimum prove fixed local NTFS, non-sparse/non-compressed attributes and reopened allocation size, and still must not be described as a universal hardware/power-loss guarantee. [Sparse Files](https://learn.microsoft.com/windows/win32/fileio/sparse-files) [Reading and Writing From a File View](https://learn.microsoft.com/windows/win32/memory/reading-and-writing-from-a-file-view)
- NTFS quotas are per-volume/per-user, charged to the owning SID using the lengths of all data streams. A hard quota can deny additional space, but it is not an attempt-directory quota unless a separately qualified SID/root topology makes it applicable. [Disk Quota Limits](https://learn.microsoft.com/windows/win32/fileio/disk-quota-limits) [System-level Administration of Disk Quotas](https://learn.microsoft.com/windows/win32/fileio/system-level-administration-of-disk-quotas)
- Sparse-file quota is based on nominal file size rather than physical allocated bytes. Thus quota usage cannot prove physical reserve allocation. [Sparse Files and Disk Quotas](https://learn.microsoft.com/windows/win32/fileio/sparse-files-and-disk-quota)
- FSRM can impose hard quotas on a folder subtree, but it is a Windows Server File Server Resource Manager capability and supports NTFS, not ReFS. It cannot be assumed present on the launch Windows desktop host. [FSRM overview](https://learn.microsoft.com/windows-server/storage/fsrm/fsrm-overview#features) [Create a Quota Template](https://learn.microsoft.com/windows-server/storage/fsrm/create-quota-template)
- Job write-byte limits in `JOBOBJECT_NOTIFICATION_LIMIT_INFORMATION` are notifications: after notification, processes continue and can write beyond the threshold. Job I/O-rate controls concern bytes per second/IOPS, not total durable storage, and the documented `SetIoRateControlInformationJobObject` surface is marked unsupported on Windows 10 version 1607 and newer. Neither is an attempt-directory hard quota. [Job notification limit remarks](https://learn.microsoft.com/windows/win32/api/winnt/ns-winnt-jobobject_notification_limit_information#remarks) [Job I/O rate control](https://learn.microsoft.com/windows/win32/api/jobapi2/ns-jobapi2-jobobject_io_rate_control_information#members) [SetIoRateControlInformationJobObject](https://learn.microsoft.com/windows/win32/api/jobapi2/nf-jobapi2-setioratecontrolinformationjobobject)
- Enforceable Job limits do exist for active-process count and committed memory; exceeding the job memory limit makes the commit fail. `KILL_ON_JOB_CLOSE` is enforceable under its last-handle condition. In contrast, a Job time limit measures user-mode execution time, not wall time, supporting a controller monotonic wall deadline instead. [JOBOBJECT_BASIC_LIMIT_INFORMATION](https://learn.microsoft.com/windows/win32/api/winnt/ns-winnt-jobobject_basic_limit_information#members) [JOBOBJECT_EXTENDED_LIMIT_INFORMATION](https://learn.microsoft.com/windows/win32/api/winnt/ns-winnt-jobobject_extended_limit_information)

These sources support v3.24's honest mandatory mode: `accounting_kill_audit` can detect, terminate, drain and block publication, but cannot promise pre-write denial or zero transient overrun. `HardWriteQuotaCapability` must remain optional and Full-Q0-qualified for the exact host primitive, volume, SID, roots, charge unit and bypass matrix.

## 6. uv exact/offline/build semantics

- `uv sync` is exact by default and removes extraneous packages; `--inexact` preserves them. The official exception is `--no-build-isolation`, where uv avoids removing possible build dependencies. [uv sync CLI](https://docs.astral.sh/uv/reference/cli/#uv-sync) [Handling extraneous packages](https://docs.astral.sh/uv/concepts/projects/sync/#handling-of-extraneous-packages)
- `--locked` asserts that `uv.lock` remains unchanged and errors if it is missing or stale relative to project metadata. It does not prove all repository, manifest or cache bytes. [uv sync CLI: `--locked`](https://docs.astral.sh/uv/reference/cli/#uv-sync) [Automatic lock and sync](https://docs.astral.sh/uv/concepts/projects/sync/#automatic-lock-and-sync)
- `--offline` disables network use and relies on cached/local inputs; `--no-python-downloads` only disables automatic Python downloads. Both are needed for the stated preparation boundary, and missing cache data may still fail. [uv sync CLI](https://docs.astral.sh/uv/reference/cli/#uv-sync) [Disabling automatic Python downloads](https://docs.astral.sh/uv/concepts/python-versions/#disabling-automatic-python-downloads)
- `--python` accepts versions, ranges, implementations, executable paths/names or installation directories and selects the first compatible interpreter discovered. A request such as `3.11` does not prove exact executable/patch identity; the manifest should bind an exact request/path and the child must read back executable and distribution identity. [Requesting a Python version](https://docs.astral.sh/uv/concepts/python-versions/#requesting-a-version) [Python discovery](https://docs.astral.sh/uv/concepts/python-versions/#discovery-of-python-versions)
- `uv run --no-sync` skips environment synchronization and implies `--frozen`; official docs state it does not check whether the environment is up to date. It is a no-mutation daily gate only after separately evidenced preparation, not an identity verifier. [uv run CLI](https://docs.astral.sh/uv/reference/cli/#uv-run) [Automatic lock and sync](https://docs.astral.sh/uv/concepts/projects/sync/#automatic-lock-and-sync)
- `uv build` explicitly supports `--build-constraint constraints.txt --require-hashes`. Constraints limit actual build requirements rather than introducing new dependencies; hash mode requires exact pins/direct URLs with matching hashes. These are `uv build` options, not `uv sync` options. [Build constraints](https://docs.astral.sh/uv/concepts/projects/build/#build-constraints) [uv build CLI](https://docs.astral.sh/uv/reference/cli/#uv-build)
- Build constraints and hashes validate build requirements, not the resulting wheel/sdist hash or reproducibility. The backend determines artifact contents/name; clean-root double-build member manifests and artifact-hash comparison remain necessary. [Building distributions](https://docs.astral.sh/uv/concepts/projects/build/#building-distributions)

## 7. Contract decisions supported by the sources

1. Keep `STARTUPINFOEXW + JOB_LIST + HANDLE_LIST + STARTF_USESTDHANDLES + CREATE_SUSPENDED`, with exact read-back/Q0 and explicit minimum Windows version.
2. Amend the successor environment acceptance to the two-stage proof in section 1; do not claim pre-resume child self-observation.
3. Treat EOF as a proof over complete writer-handle ownership and closure, not as a consequence of process exit alone.
4. Keep `accounting_kill_audit` mandatory and `HardWriteQuotaCapability` optional; Job write notifications/rate control are not storage quotas.
5. Build the 1 GiB reserve with allocation+EOF alignment, non-sparse/non-compressed checks and close/reopen allocation read-back; label stronger physical/durability claims as Full-Q0 host evidence, not API fact.
6. Use current-user DPAPI without `LOCAL_MACHINE`, plus distinct purpose material and application integrity; keep restore eligibility/single consumption in the state contract.
7. Keep uv environment preparation separate from gates, use `run --no-sync` only after preparation evidence, and retain child identity read-back plus repeated clean-root build comparison.

## 8. Claims the official material does not support

- A suspended child can self-observe or report its environment before `ResumeThread`.
- A bounded EOF deadline follows automatically from Job termination without proving every write-handle owner.
- `KILL_ON_JOB_CLOSE` works while any unknown Job handle remains open.
- Leading-`=` pseudo variables are ordinary documented environment-variable names; the policy should simply reject them as non-catalog input.
- DPAPI description/entropy alone proves the project's purpose, generation, rollback or single-consumption semantics.
- `GetDiskFreeSpaceEx`, Job I/O notifications/rate control, NTFS per-user quota or desktop Windows alone supplies an attempt-directory atomic hard-write quota.
- File EOF, logical size or NTFS quota charge alone proves a durable fully allocated physical reserve.
- `--locked`, `--offline`, `--no-python-downloads`, `--python 3.11` or `--no-sync` individually proves the complete runtime composition identity.
- Hashed build requirements alone prove deterministic final artifacts.
