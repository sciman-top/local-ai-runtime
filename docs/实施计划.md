当前执行顺序：

1. `P0-1`：`cap_drop:[ALL]` 已在旧 `official_root_bootstrap` 历史线和新 accepted replacement runtime model 上都完成真实探针；当前结论一致，都会打坏真实 `docker run` 路径，因此现阶段应固定为 `still blocked / not accepted`，而不是继续写成“等待运行模型变化后再评估”
2. `P0-2`：已完成 repo-side 隔离真实探针，最新结论见 [phase0-readonly-rootfs-probe-20260628.md](D:/CODE/hermes-agent/docs/phase0-readonly-rootfs-probe-20260628.md)
   - `/run` 不能使用默认 tmpfs；当前最小必要条件是 `--tmpfs /run:exec` 与 `--tmpfs /tmp`
   - 当前 accepted 运行模型 `official_root_bootstrap + 10001:10001 remap` 在 `read_only rootfs` 下仍稳定阻塞
   - comparative probe 只改 runtime uid/gid 为 `10000:10000` 后，`usermod` 与 `/opt/data` 权限错误已消失，容器可越过 bootstrap / reconcile 阶段
   - retained volume follow-up 已证明：显式 `--provider openai-api --model gpt-5.4` 的 `--oneshot` 可返回 `OK`
   - 本轮又用 probe-only fresh volume config bootstrap 把默认 `--oneshot` 也跑到了 `OK`
   - 配套对照已确认：关掉 bootstrap 时 `10000:10000` 会回到 `hermes -z: no final response was produced`；回到 accepted `10001:10001` 时即使保留 bootstrap 仍稳定失败于 `usermod + /opt/data` 权限链
   - 纯 `derived_non_root` 派生镜像如果直接以 `10001:10001` 启动容器，会更早失败在 `s6-overlay preinit` 的 `/run` 所有权检查
   - repo-side 混合候选运行模型 `derived image + runtime_user 10001:10001 + container_start_user 0:0` 已完整通过隔离 `bring-up / boundary / snapshot`

## 变更纪律

以后所有升级都按这个顺序：

1. 先读当前 known-good
2. 只改一个变量
3. 重跑契约检查
4. 重跑 bring-up gates
5. 重跑 boundary 验证
6. 生成新 snapshot
7. 记录差异、风险、回滚点
