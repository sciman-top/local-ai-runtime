## Closed Decisions

### Environment And Config

- Every Git process starts from a hardened empty Git environment; inherited `GIT_*`
  variables and environment-dependent config are rejected.
- Config classification is deny-by-default. Includes, credential/HTTP/proxy/header,
  executable filters/drivers/hooks, signing/SSH, rewrites, protocol/submodule, aliases,
  editors/pagers and unknown keys are blocked.
- Name-only remote and LFS keys are classified without reading their values. Safe values
  must match the complete versioned `GitConfigPolicy.v1` catalog.

### Canonical Objects And Commit

- Claim CAS generates `claim_epoch_seconds` once and binds it through the immutable
  `ResolvedAttemptManifest`, `GitObjectPlan` and commit bytes.
- The controller independently frames canonical blob/tree/commit payloads and computes
  their SHA-1 OIDs; Git is a cross-check, not the only oracle.
- Existing objects are accepted only when type, declared size, payload bytes and computed
  OID all match. Loose zlib representation is never used as an equality oracle.
- A commit has exactly one parent and fixed `tree`, `parent`, `author`, `committer` header
  order. Author and committer share the claim time and fixed identity encoding.

### Publication And Cleanup

- Objects are materialized in an attempt-local object directory and promoted only after a
  sealed `ObjectSetManifest` matches the complete canonical graph.
- Publication order is fixed as promote -> reachability -> index -> HEAD -> task ref ->
  evidence -> remove. Finalization proves path/mode/bytes equality before metadata moves.
- HEAD and task-ref updates use `--no-create-reflog`; task ref uses zero-OID CAS. Historical
  successful task refs are durable and never deleted.
- Cleanup is ownership-scoped and cannot use `reset --hard` or delete unknown paths.
