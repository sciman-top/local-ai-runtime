# Local AI Runtime 0.2 v3.18：Unified Native + Batch Deterministic Minimum-Operator Implementation Baseline Candidate

## 1. 规范身份、产品终态与批准边界

- 唯一规范 ID 为 `local-ai-runtime-0.2-v3.18`。本文是自包含候选基线；实现不得引用 v3.14-v3.17 或历次评审补全语义。任何正文、schema、catalog、transition row 或安全默认值变化都必须创建新版本。
- 版本谱系固定为：v3.14 是 `canonical_predecessor`，SHA-256 为 `B4133FF27E1FACD0B2B8C48BB89D5FDC4006AD379203BFEF78B4AF6CDAC9DDB2`；v3.16 是 `withdrawn_candidate`，SHA-256 为 `6EAF5320495247661052974333023B1131A1B12A6BFD60BD730975489BB1A9ED`；v3.15 是无 canonical hash 的 `withdrawn_draft`。
- v3.17 是 `superseded_candidate`。当前 `A285F5F421A8CCD4DEBD8794609A2AA0EB07BB1BF651C2467A95F7CAD25A5F81` 仅为 32,825 字节对话正文的 `provisional_transcript_hash`；须先归档完全相同字节，再由 pinned Python `hashlib` 与 PowerShell `Get-FileHash` 独立复算，方可写入 manifest。若结果不同，记录差异并采用实际归档 hash，禁止静默改写正文。
- 规范正文只接受字节格式验证：UTF-8、无 BOM、LF、Unicode NFC、恰好一个末尾 LF、无 CR/NUL/未批准控制字符、无行尾 SP/HTAB；Markdown hard break 使用 `<br>`。验证器不得转换编码、Unicode、空白或换行；SHA-256 直接计算验证后的原始字节，正文不嵌自身 hash。
- `ReviewEvidenceIndex.v1` 是正文外独立、append-only 的 hash chain；评审文件只能作为 `review_provenance`，不得充当规范前身。`BaselineApprovalRecord.v1` 固定规范 hash、BaselineManifest hash 和批准时的 review head；撤回使用不可变 `BaselineApprovalRevocationRecord.v1` 与 active-approval generation CAS，后续评审不能修改既有批准记录。
- 三层门固定为 `Baseline Approval -> Implementation Acceptance -> Full Q0 / P2 Admission`。批准前 `blocking_stage=baseline_approval`；批准后、代码验收前为 `implementation_acceptance`；代码验收后、Q0 前为 `full_q0`。Q0 只阻断 P2，不阻断已批准基线下的 P0/P1 实现。
- 产品终态固定为：Codex Native Direct/Spec/Program + Python Policy/Evidence Kernel + 全局单 writer、commit-only Batch + 最终只读 legacy/Hermes/AgentBridge compat。词典序目标为：安全和副作用正确性硬门、最小净人工分钟、提高无人值守成功率、再优化周期和可归属成本。
- Native Direct 用于明确的一次性交互任务；Native Spec 最多询问三个关键问题并产出 decision-complete TaskSpec，不自动执行；Native Program 仅用于至少两个独立写集合且集成顺序已固定的任务，可使用原生 subagents/worktrees，但由人工集成且不继承 Batch receipt/fence。
- Batch 只接受已批准模板、repo/template qualification、有效 Authorization 和显式 `BatchSubmission`。自由 prompt 永不自动转 Batch；Spec-to-Batch 必须先形成版本化 `TaskTemplate`，再由操作者提交。
- GUI、生产、数据库迁移、VPS、凭据操作、remote、不可逆或高风险任务始终走 Human-controlled Native。Batch 0.2 禁止多 writer、subagent、Approval、SDK、多 Provider、动态 fallback、依赖安装、task-side 网络、remote Git、merge/push、target-ref 更新和 task-ref 删除。
- Batch 唯一交付为本地 deterministic commit 与 `refs/heads/codex/batch/<task_uuid>-a<attempt_no>`；人工负责最终检查、集成、merge/push。正常 Batch 零干预；人工仅处理 login、environment/repo/template qualification、Authorization、maintenance、cutover/rollback 和明确的 recovery/repair。
- 当前仓库真实内核仍是 `runtime/host-orchestrator`，active queue 仍声明 `PHASE-1-VERTICAL-SLICE`，当前 selector 因 planning verifier 缺口返回 `repair_gate_first`。Truth Reset 完成前不得宣称新 Batch 已实现、platform compatible 或 live accepted。

## 2. 契约、输入、Codex 与隔离边界

- P0 后的新包归宿为 `runtime/local-ai-runtime/src/local_ai_runtime/`，采用 Python 3.11+ 模块化单体，模块边界为 `contracts/kernel/qualification/storage/execution/recovery/git_local/compat`；新包不得 import、调用或双写 `host_orchestrator`。Legacy 与新 Batch 仅共享版本化 ownership wire contract、repo mutex 和 generation，不共享数据库。
- Tier A 必须提供 `Spec + Schema + Example + Catalog + Verifier`，至少包括 `BaselineManifest/ProductContract/WorkRoutingPolicy/CanonicalizationPolicy/RepoProfile/TaskTemplate/BatchSubmission/ResolvedBatchManifest/Authorization/GatePolicy/RuntimeActivationRecord/RuntimeToolchainManifest/QualifiedEnvironmentBinding/CodexFeaturePolicy/ProcessEnvironmentPolicy/GitConfigPolicy/ExecutionReceipt/EvidenceIndex.v2/CloseoutBundle.v2/WorkflowEffectMetrics.v1`，以及全部状态和治理 policy。
- 状态和治理 Tier A 固定拆为 `SubmissionFamilyStatePolicy.v1`、`BatchTaskStatePolicy.v1`、`AttemptStatePolicy.v1`、`PlatformOperationalStatePolicy.v1`、`RepoCutoverMaintenancePolicy.v1`、`TemplateLifecyclePolicy.v1`、`AutonomyPolicy.v1`、`OperatorActionCatalog.v1` 和 `GuardCatalog.v1`；禁止在命令处理器中补隐藏分支。
- Tier B 至少包括 `QualificationSensitiveInputSet/RepoTemplateQualification/ResolvedAttemptManifest/AttemptRecord/AttemptProcessEnvironment/InstructionSnapshot/SkillInventory/FeatureInventory/AuthState/AttemptPermissionOverlay/WriterFinalResult/NormalizedExecutionEvent/JournalSegmentManifest/GateReport/GitPreflightReport/TaskResolution/OperatorWorkSession/RepoMaintenanceRecord/JobIdentity/AttemptAuthorizationContinuation` 及 fenced action、object set、artifact/outbox、backup、cutover记录。
- Tier C 只保留内存类型和测试 fixtures。批准模板源码可以作为 immutable policy artifact 存在，但渲染后的 prompt/stdin、raw Codex JSONL、stdout/stderr、reasoning、agent text、tool I/O、argv、env 和 config dump永不持久化；raw stdout/stderr 的内容 hash 也禁止持久化。
- `CanonicalizationPolicy.v1` 对 JSON 固定为：解析阶段拒绝重复 key；字符串必须已是 NFC；禁止浮点；整数由 schema 有界；对象 key 按 UTF-8 字节词典序；无多余空白并使用最小 JSON escaping。数组默认保序，只有 schema 明确声明 `set_semantics` 和唯一 sort key 时才排序；重复 set key直接拒绝，不静默去重。
- 所有可复算对象使用 domain-separated envelope：`{"domain":"local-ai-runtime/<type>/v1","schema_version":1,"payload":...}`，hash 为 canonical UTF-8 envelope 的小写 SHA-256。对象自身的 hash 字段不进入其 payload。
- Git path 必须保留 Git 原始 UTF-8 拼写、大小写和 `/`，只验证而不执行 NFC、大小写或分隔符改写；拒绝非法 UTF-8、非 NFC、绝对路径、`.`/`..`、ADS、DOS device、未批准控制字符和 Windows case collision。NTFS collision key 由版本化 Windows invariant-uppercase 算法独立生成，不替代 Git path。
- `QualificationSensitiveInputSet.v1` 使用完整 union：`present` 记录 repo path、host key、object type、mode、OID、rule/source/class；`absent` 记录候选路径及 base-tree absence proof；`expanded` 记录目录规则及有序 child-entry hashes；`blocked` 记录 rule/source/reason 并使 qualification 失败。Fingerprint envelope还绑定 repo identity、唯一 `local_base_ref`、base commit/tree、resolver/catalog版本及完整有序 entry union。
- 输入发现只从 expected base Git tree取证。命中敏感路径、发现入口或其祖先的 dirty、staged、untracked 状态一律阻断，不把工作树 hash 混入 Authorization；repo声明只能扩大输入集合。未知动态加载、repo外引用、递归闭包不完整或 collision 均阻断。
- `TaskTemplate.parameters` 必须使用 `additionalProperties=false`，只允许显式有界的 enum、bool、integer、ID、approved-relative-path 及其有界数组；禁止任意字符串或自由对象。Canonical parameters 总量不超过 64 KiB，单路径同时受 schema、UTF-8/UTF-16 和 repo path policy 约束，并在持久化前通过 credential/secret catalog。
- 参数不得直接或经字符串插值影响 executable、argv、environment、permission roots、gate集合、Git ref、model/provider、feature、sandbox、Authorization 或 policy generation。模板编译器只把参数编码为固定 schema 的 canonical data block；渲染后的 writer stdin只存在于有界内存。
- `BatchSubmission` 仅含 `repo_id/template_id/parameters/expected_base_commit`；repo/template ID 匹配 `[a-z0-9][a-z0-9._-]{0,62}`，base 为 40 位小写 SHA-1，task/attempt 为 controller 生成的小写 UUID。Domain-separated submission fingerprint 是永久 family key，普通重复 submit 永远返回 generation 0 `root_task_id`。
- 合法重提只允许 `batch resolve <source_task_id> --code create_resubmission_v1`。单个 `BEGIN IMMEDIATE` 事务必须先查询既有 `TaskResubmission` 并重放 successor；否则验证 source 为 family current、`failed|cancelled`、terminal snapshot未变、Job死亡、journal/evidence已封存、actions均 terminal/quarantine、全 family 历史从未成功创建 task ref，并重验 base、qualification、Authorization 和 environment，然后写 immutable `TaskResolution.v1`、创建 generation+1 successor、写 `TaskResubmission(source_task_id PRIMARY KEY, successor_task_id UNIQUE, resolution_id UNIQUE)` 并 CAS `SubmissionFamily.current_task_id`。
- `UNIQUE(root_task_id,resubmission_generation)` 防止代际冲突；重复命令必须在“source仍为current”检查前返回已有 successor。原 task 永不修改；completed、已有或历史成功 task ref 的 family 永不重提。Stale base 必须重新 prepare/submit并形成新 fingerprint；不引入 ResubmissionPermit、Consumption 或 `batch resubmit`。
- Authorization 默认有效 30 天，绑定 repo/template、submission/frozen manifest、sensitive fingerprint、toolchain/environment/model/profile/policy generations和允许动作。Claim 时剩余期限必须不小于 attempt deadline+5分钟，并在 submit、claim、spawn、heartbeat、verify、promotion、finalize、task-ref、evidence和cleanup前复验。
- 运行根固定为 `%LOCALAPPDATA%\LocalAIRuntime`：`versions/policies/schemas/codex-home` 为内容寻址、只读共享；`state/ownership/evidence/backups/quarantine` 为controller管理；`environments/<binding_hash>` 为只读依赖；每个 attempt 独占 `runs/<attempt>/profile/xdg-config/codex-sqlite/tmp/cache/spool/worktree/empty`。
- 环境机械映射为 `CODEX_HOME=<immutable codex-home>`、`HOME=USERPROFILE=<attempt>/profile`、`APPDATA=<profile>/AppData/Roaming`、`LOCALAPPDATA=<profile>/AppData/Local`、`XDG_CONFIG_HOME=<attempt>/xdg-config`、`CODEX_SQLITE_HOME=<attempt>/codex-sqlite`、`TEMP=TMP=<attempt>/tmp`、所有 writable cache 到 `<attempt>/cache`。不同 attempt 不得共享任何可写 HOME、SQLite、APPDATA、TEMP、cache 或 spool。
- 每次 writer/gate/Git spawn 前后都要验证 writable roots及全部祖先的 final path、volume/file identity、owner/DACL、reparse point和文件 hardlink count；任何漂移阻断。正式运行只支持通过 no-replace、`ReplaceFileW`、FlushFileBuffers 和 crash probes 的本地固定 NTFS 卷，不声称存在通用 Windows parent-directory fsync。
- `RuntimeToolchainManifest.v1` 锁定 Python、uv、`uv.lock`、PowerShell、Git、Node/npm wrapper、`@openai/codex@0.144.1`、bundled `codex.exe`、Windows build、sandbox、Job helper、bootstrap、CapabilityAdapter 和 Q0 probes 的绝对路径、版本、file identity及SHA-256；repo专用依赖只进入 immutable `QualifiedEnvironmentBinding`，Batch 不 install/restore/bootstrap。
- 版本安装到 `versions/<version>`；bootstrap 使用受管 PowerShell `-NoProfile -NonInteractive` 和 pinned Python `-I -s -E`。`current.json` 首次 no-replace，升级/回滚使用 expected generation/checksum、flush、atomic replace及backup，并由 `RuntimeActivationRecord.v1` 约束 schema compatibility。
- Writer 命令固定为 `<codex> exec <overlay> --ephemeral --json --strict-config --ignore-rules -p batch -C <worktree> -m gpt-5.6-sol --output-schema <WriterFinalResultSchema> -`；gate/Git固定为 `<codex> sandbox -p batch -P <batch-gate|batch-git-audit|batch-git-local> -C <cwd> <overlay> -- <absolute-executable> <argv...>`。0.2 不做 model/provider/sandbox fallback。
- Writer profile 固定 `default_permissions="batch-writer"`、`model_reasoning_effort="high"`、`approval_policy="never"`、`windows.sandbox="elevated"`、`web_search="disabled"`、`history.persistence="none"`、`allow_login_shell=false`、`shell_environment_policy.inherit="none"`、`cli_auth_credentials_store="keyring"`；overlay无条件生成 `projects."<canonical-worktree>".trust_level="untrusted"`，并绑定 compiler input、safe projection、effective config、quick preflight和Q0。
- `CodexFeaturePolicy.v1` 将每项分类为 `required_enabled/allowed_internal/required_disabled/removed_ignored` 并绑定 model-visible tool inventory。apps/plugins/hooks/MCP/memories/goals/multi-agent/browser/computer-use/image generation/tool suggestions/workspace dependencies/shell snapshot/skill install等必须不可见；未知 effective=true、removed能力复活或inventory漂移均为 incompatible。
- `project_doc_fallback_filenames=[]`，global AGENTS不超过8192字节，project chain不超过32768字节；`InstructionSnapshot`绑定全局文件absence/hash、完整project链顺序/OID/bytes及skill inventory。`untrusted`不替代AGENTS、skills、repo内容和Git surface审计。
- 0.2 仅支持 keyring；不可用即Q0失败，不提供 file/auto fallback。`AuthState.v1`只保存 generation、store、status和qualified time；`needs_auth`全局停止新的provider claim，按 `(platform_login_required,auth_generation)`去重通知，只允许交互式login改变generation；controller-only drain、seal、reconcile和cleanup仍可运行。
- Codex parent只获得provider控制面网络和Q0批准的非秘密proxy endpoint；writer工具、gate和Git进程不得获得auth/proxy变量或task-side网络。生产中任何 denied-network-attempt 使该任务失败并暂停对应模板；任何成功的未授权egress是全局安全硬失败。
- Overlay最多64个 `-c`、16,384 UTF-16 code units，完整CreateProcess命令行小于32,767；PATH只含 ToolchainManifest/EnvironmentBinding目录。Raw argv/env不落盘，只保存compiler版本/hash、逻辑根替换后的安全投影、有序digest、effective-config hash和inventory/probe IDs。
- 限额固定为 writer 3600秒、单gate 1800秒、全部gates 7200秒、closeout 900秒、attempt 11700秒、scheduler 12600秒；writer JSONL/stdout合计8MiB、stderr 8MiB、单行256KiB、normalized journal总计8MiB、单event 16KiB、final result 1MiB，其他进程stdout/stderr各8MiB、diff 16MiB、changed files 200、artifacts 256MiB；`max_attempts=3` 按 task generation计算。
- 活进程deadline使用 monotonic clock，耐久时间使用UTC微秒；记录boot identity和wall-clock rollback检测，重启或时钟异常使lease失效并进入recovery。Claim前磁盘必须至少为 `max(5 GiB, 2*base materialized bytes + 2*artifact limit)`。

## 3. Ownership、writer、journal、fencing 与 Git 提交协议

- `ownership/<repo_id>.json` 绑定repo ID、canonical Git common-dir final path、volume/file ID、owner、status、generation、registry generation、schema和checksum；首次 no-replace，更新使用 expected generation/checksum、flush、atomic replace和backup。损坏时只可采用identity一致且generation不倒退的backup，否则要求交互式ownership repair。
- SID canonical form来自 `ConvertSidToStringSidW`；`SIDHash` 为 ASCII SID 的SHA-256前16个小写hex字符，`RepoIdentityHash` 为domain-separated canonical repo identity SHA-256前24个字符。名称固定为 `Global\LocalAIRuntime.BatchDrain.<SIDHash>.v1`、`Global\LocalAIRuntime.RepoOwnership.<SIDHash>.<RepoIdentityHash>.v1`、`Global\LocalAIRuntime.Attempt.<SIDHash>.<attempt_uuid>.v1` 和 `Global\LocalAIRuntime.Job.<SIDHash>.<attempt_uuid>.v1`。
- Named objects 的owner/group、type、DACL、name length和Job limits必须核验；SDDL固定 `D:P(A;;GA;;;SY)(A;;GA;;;<CURRENT_SID>)`。Legacy/new必须消费同一repo名称算法。既有对象不匹配时进入 `platform_incompatible`，禁止仅凭名字复用。
- 锁顺序固定为 global capacity -> repo mutex/ownership generation -> attempt mutex -> SQLite `BEGIN IMMEDIATE` CAS。Heartbeat 15秒、TTL 90秒；takeover同时证明旧controller mutex释放、PID+creation time死亡、TTL到期、Job状态可解释并完成fence CAS，且优先恢复旧attempt。
- 同名Job即使零进程也不得复用。检查和重建必须全程持有attempt mutex：OpenJobObject验证type/owner/DACL/limits/process list，关闭检查handle，再调用CreateJobObject；若返回 `ERROR_ALREADY_EXISTS`，进入 `recovery_pending/job_handle_still_open`。禁止换名、复用旧handle或忽略对象；只能终止经 executable hash、PID、creation time证明属于runtime的进程，否则等待持有者退出或重启宿主。
- Writer marker固定为 `<attempt>/writer-start.v1`，内容绑定task generation、attempt、fence、launch-intent hash和checksum，使用 `CREATE_NEW|FILE_FLAG_WRITE_THROUGH`、FlushFileBuffers、close，并由NTFS power-loss probe验证。启动状态唯一合法链为 `launch_intent -> durable_marker -> spawned_suspended -> writer_execution_committed -> resume_observed|resume_outcome_unknown -> process_exited`。
- Spawn必须使用 `CREATE_SUSPENDED + PROC_THREAD_ATTRIBUTE_JOB_LIST` 原子加入预建Job，禁止breakaway；持久化PID及creation time后才进入 `spawned_suspended`。`writer_execution_committed` 必须在 `ResumeThread` 前事务提交并flush。
- 只有 `launch_intent|durable_marker` 且证明Job/PID不存在时，才可继续同一launch intent。一旦 `spawned_suspended`，同一attempt不得创建第二个writer；若能证明进程从未execution-commit且已终止，可创建新attempt。一旦 `writer_execution_committed`，当前task generation永久禁止任何新writer，即使零tool/零mutation或resume结果未知；只有通过显式resubmission创建的新task generation才有自己的唯一writer。
- `FencedActionIntent.v1` immutable，`FencedActionAdoption.v1` append-only，`FencedActionHead.v1` mutable CAS，`FencedActionResult.v1` immutable terminal。创建intent时同时创建head，指向 `kind=intent/hash=intent_hash`；intent永不修改。
- Adoption字段至少包括 `action_id/prior_head_kind/prior_head_hash/prior_fence/new_fence/expected_result_hash/dead-controller-or-job-proof/adoption_hash`。插入adoption和CAS更新head必须在同一 `BEGIN IMMEDIATE` 中；`prior_head_hash NOT NULL`、`UNIQUE(action_id,prior_head_hash)`、head主键唯一、result按action唯一。写result前重验head hash/fence/version和预期结果；terminal后永久禁止继续收养。
- 可收养动作仅限结果完全确定的controller动作：`create_worktree/checkout_base/materialize_object_set/artifact_publish/promote_objects/finalize_worktree_index/finalize_worktree_head/create_task_ref/remove_worktree`，以及绑定完整JobIdentity的 `terminate_job`。Writer、自由命令、auth refresh、qualification和需要重新决策的动作永不可收养。
- `AttemptAuthorizationContinuation.v1` 必须绑定attempt、旧/新Authorization ID与generation、旧/新fence、prior continuation hash、frozen manifest、writer marker、terminal receipt、environment/toolchain hashes和明确允许阶段 `verify|publish|evidence|cleanup`；采用append-only链和head CAS。它只允许继续既有attempt，永不允许writer重跑。
- Writer/gate/Git stdout和stderr必须由独立异步reader持续、有界排空，避免pipe deadlock；正文和content hash立即丢弃，只保留byte count、exit code、reason code和白名单结构化结果。输出超限默认只失败当前attempt，除非独立Q0证明平台行为漂移。
- `NormalizedExecutionEvent.v1` 公共必需字段为 schema、attempt UUID、positive fence/seq、`YYYY-MM-DDTHH:MM:SS.ffffffZ`、event_type、status、prev_hash、event_hash；optional字段不得为null。首条prev为64个零，hash为domain-separated canonical event excluding `event_hash`。
- Exhaustive event/status矩阵固定为：`process_started/tool_started -> started`；`mutation_observed/stream_eof -> observed`；`content_validated/tool_completed/turn_completed/final_result/journal_sealed -> completed`；非预期 `process_exited`、`tool_failed/resource_limit_exceeded/process_terminated -> failed`；`adapter_rejected -> rejected`；预期零退出且framing完整的 `process_exited -> completed`。未知type/status组合退出2并关闭Job。
- `mutation_observed` 在扫描前只允许CSPRNG `mutation_observation_id`、byte_count、枚举path_class，以及映射成功时公开、版本化的 `approved_path_id`；未知、deny、越界或reparse路径不得保存原文、无key hash或可字典枚举digest。路径和内容均通过policy与secret scan后，后续 `content_validated` 才可携带canonical relative path和content SHA-256；失败只记录随机quarantine ID、byte count和reason code，既有event永不修改。
- Tool事件必须有item_id；failed/rejected必须引用versioned reason-code catalog；exit_code只允许tool/process终结；usage只允许turn_completed。`WriterFinalResult.v1` 只允许 `completed|blocked`、最多200个排序去重的approved path及可选枚举reason_code，无summary或自由文本，且controller仍以磁盘/Git审计为准。
- Journal写入顺序固定为 append segment -> FlushFileBuffers -> SQLite cursor事务；DB只能落后journal。`JournalSegmentManifest` 固定 `segment_no/previous_segment_hash/accepted_end_offset/first_seq/last_seq/seal_hash`，continuation从最后有效event hash继续；损坏不截断，使用随机ID的ACL受限sealed quarantine保存，公开evidence不含内容派生hash。
- 完整换行但非法JSON/schema/event、未知schema或正常process exit后的partial line表示CLI/adapter incompatibility；cancel、timeout、controller kill或异常崩溃留下的EOF partial buffer仅属于attempt failure/recovery，不暂停平台。
- `ExecutionReceipt` 只有在 process exit、JSONL EOF、final schema、全部limits、完整segment/event-chain seal和Job零进程六项全部成立时发布；receipt verifier是硬不变量，并绑定manifest、config、feature/tool inventory、reason catalog、writer marker和journal chain，不保存raw output hash。
- SQLite固定 `journal_mode=DELETE`、`foreign_keys=ON`、`synchronous=FULL`、`busy_timeout=5000`和短事务。表必须覆盖families/tasks/resolutions/resubmissions/attempts/leases/writer markers/events/segments/artifacts/outbox/fenced action四表/object sets/authorizations/continuations/environments/qualifications/runtime activations/operator actions/work sessions/platform controls/repo maintenance/cutover/template lifecycle/backups；所有副作用绑定attempt和fence。
- Artifact发布固定为已扫描的immutable spool temp写入+flush -> outbox/fenced intent -> no-replace publish -> read-back size/hash -> immutable terminal。Intent前孤儿temp进入随机ID quarantine且不自动收养；已存在final只在同intent/hash/size时确认。Raw process输出永不进入artifact。
- `RepoProfile.evidence_mode` 必须显式为 `runtime_external_v1` 并绑定外置evidence root的final identity；这是P2 writer qualification硬门。Commit message不含evidence ID；`ExecutionReceipt/EvidenceIndex.v2/CloseoutBundle.v2` 在commit/ref后建立关联，避免commit与evidence内容寻址环。
- 第一次Git调用前，controller以Win32 no-follow解析 `.git` file/common-dir并审计路径身份；第一条Git命令必须已位于final env、Job、overlay和 `batch-git-audit`，固定执行 `git --no-pager ... rev-parse --path-format=absolute --git-common-dir`。所有后续Git使用absolute executable/argv且不经shell。
- Git环境固定 `GIT_CONFIG_NOSYSTEM=1`、`GIT_CONFIG_GLOBAL=NUL`、`GIT_CONFIG_SYSTEM=NUL`、`GIT_ATTR_NOSYSTEM=1`、`GIT_TERMINAL_PROMPT=0`、`GIT_OPTIONAL_LOCKS=0`、`GIT_NO_REPLACE_OBJECTS=1`，清除其他继承 `GIT_*`。`GIT_CONFIG_COUNT=8` 依次固定 `core.hooksPath=<managed-empty>`、`core.excludesFile=NUL`、`core.attributesFile=NUL`、`commit.gpgSign=false`、`core.fsmonitor=false`、`core.logAllRefUpdates=false`、`gc.auto=0`、`maintenance.auto=false`；operation-scoped object/index变量只能由adapter设置。
- Git config默认deny。include、credential/http/proxy/header、filter、diff/textconv、merge driver、hooks、signing、fsmonitor、SSH、URL rewrite、protocol/submodule、alias/editor/pager、外部attributes/excludes及未知key全部阻断；remote/LFS等潜在URL只允许保存key name，不保存value。审计 `.git/info/attributes`、`.git/info/exclude`、hooks、config.worktree、alternates、replace/grafts、base tree中的 `.gitattributes/.gitmodules/.lfsconfig`、touched-path attributes及protected path/mode/OID。
- 总执行链固定为 ownership/fence -> original-repo audit -> fenced worktree/checkout -> writer -> writable-root和protected-surface复核 -> secret scan -> offline gates -> attempt-local index/blob/tree/commit -> seal `ObjectSetManifest` -> promote objects -> clear alternates后的common-store reachability -> finalize index -> finalize detached HEAD CAS -> task-ref CAS -> evidence -> remove_worktree。
- Changed blobs、trees和commit只先写attempt-local `GIT_OBJECT_DIRECTORY`，base common objects仅作为只读 alternate。`ObjectSetManifest` 在common store副作用前记录输入/index hash、每个object的type/size/OID/payload SHA-256和完整reachability graph；promotion必须先有intent并create-if-absent，existing object通过canonical Git type+size+payload重新计算 `SHA1("<type> <size>\0"+payload)`，不得比较loose zlib字节。
- Commit headers严格按 `tree`、唯一 `parent=<expected_base_commit>`、`author`、`committer` 顺序，禁止额外header；message为UTF-8、无encoding header、恰好一个末尾LF。Identity固定 `Local AI Runtime <local-ai-runtime@localhost.invalid>`，author/committer时间为claim时持久化的UTC秒和 `+0000`；正文固定 `batch(<template_id>): <task_uuid> attempt <n>\n\nManifest: sha256:<resolved_manifest_hash>\n`，controller预计算expected SHA-1。
- `finalize_worktree` 禁止 `reset --hard`。先证明tracked工作树与expected tree逐path/mode/bytes一致且无未批准untracked/ignored文件，再以临时expected index、fence和old-index hash原子替换真实index并read-back；随后用 `update-ref --no-deref HEAD <expected_commit> <expected_base_commit>` 单独CAS。两步分别是可收养action。
- Task ref只在index、HEAD和common reachability均通过后执行 `update-ref <exact-ref> <expected_commit> <zero-oid>`；任何历史成功结果即使ref被外部删除也永久禁止resubmission。自动remove仅在runtime-owned、Job终止、fence一致、actions/evidence terminal且HEAD/index/worktree干净时执行；remove后核验目录和common-dir worktree admin记录消失。否则进入cleanup_required并保留worktree和task ref。
- Cancel在pre-spawn可直接terminal；running/verifying必须关闭Job、排空输出并handoff recovery；closing只记录枚举reason和 `accepted_deferred`。同task自动retry只允许无writer execution commit、零tool/mutation/GitAction、工作树等于base且attempt数未达3；其他情况只能失败或显式创建successor task。
- Quiescent backup仅是state/evidence backup：按canonical repo identity排序取得global、registry和所有被引用repo mutex，要求零active lease、零nonterminal/recovery attempt和零pending outbox；包含DB、activation、policy、ownership、terminal evidence及被terminal记录引用的sealed quarantine，不含auth、environment内容、worktree、普通quarantine或非终态spool。Restore drill只恢复到新隔离root并验证，不覆盖生产root。

## 4. 状态、治理、CLI、Native maintenance 与迁移

- 每张transition table必须真实落盘，每行包含 `source_state/operation_or_event/guard_ids/allowed_effects/target_state/exit_code/capacity_disposition/scheduler_priority/retry_policy`。`GuardCatalog.v1` 为每个guard固定scope、input snapshot hash、precedence、reason code和跨表依赖，并由verifier证明无环；未知state、operation、resolution或guard组合统一退出2。
- Guard阻断优先级固定为 baseline approval -> implementation acceptance -> P2 Q0 -> platform incompatible -> manual drain/suspend -> needs_auth -> platform unavailable/qualification suspended -> repo ownership/maintenance/requalification -> template lifecycle/authorization -> task Authorization/fingerprint/base/environment -> recovery priority -> capacity。安全drain/kill/seal使用明确的 `safety_only` guard，不构成通用旁路。
- `SubmissionFamilyStatePolicy` 唯一合法主转移为：不存在+new submit -> `open`；任意状态+ordinary replay -> 状态不变并返回root；open+current failed/cancelled且无ref历史 -> `awaiting_resolution`；open+task-ref成功但closeout未完 -> `publication_recovery`；publication_recovery+evidence/cleanup完成 -> `commit_ready`；awaiting_resolution+create_resubmission guards -> open并CAS新current；awaiting_resolution+stale/historical-ref/explicit-close -> `closed`。`commit_ready|closed` 不得重新打开。
- `BatchTaskStatePolicy` 主路径为 `submitted -> queued -> active -> closing -> completed`；queued在外部guard阻断时 -> parked，guard恢复 -> queued；active仅在RetryEligibility通过时 -> queued新attempt，否则 -> failed；pre-spawn cancel -> cancelled，active/closing cancel -> closing并deferred，安全收口后 -> cancelled。Source task在resubmission后保持原terminal状态。
- `AttemptStatePolicy` 主路径为 `created -> claimed -> preparing -> writer_launching -> writer_running -> verifying -> closing -> succeeded`；中断后仅在Job零进程、journal已seal或durable continuation、无executing action且SQLite handoff完成时进入 `recovery_pending` 并释放capacity。恢复优先进入claimed-recovery，再按checkpoint回到verifying/closing；无法自动证明时进入 `recovery_required|reconcile_required|cleanup_required`。
- Attempt resolution只允许 `continue_verify_v1/continue_closeout_v1/retry_cleanup_v1/terminate_failed_v1/create_resubmission_v1`；reconcile只读且不得创建object/ref副作用。等待auth/environment/authorization的attempt必须park、去重通知、释放capacity，不得阻塞其他repo；due recovery始终优先于新task。
- `PlatformOperationalStatePolicy` 使用独立lane而非笛卡尔积：auth=`unknown|ready|needs_auth`；control=`running|drain_requested|suspended`；qualification=`unqualified|qualified|suspended|incompatible`；availability=`available|unavailable`。Full Q0驱动qualification，login驱动auth，operator CAS驱动control；全局resume只清除control，不覆盖其他lane。
- `platform_unavailable` 按Retry-After，否则5m -> 15m -> 1h -> 6h；canary基础设施只独立重试一次，随后qualification suspended。Binary/config/profile/effective behavior漂移直接 incompatible。Scope-aware stop-the-line只阻断对应global/repo/template/attempt范围。
- `RepoCutoverMaintenancePolicy` 的ownership lane为 `legacy -> guarded_legacy -> batch`，满足零active和rollback guard时可 `batch -> guarded_legacy`，30天零legacy调用且全部repo完成后进入 `retired`；maintenance lane为 `idle -> drain_requested -> active -> idle|repair_required`，永不按TTL自动解除；qualification lane为 `qualified|repo_requalification_required`。
- `TemplateLifecyclePolicy` 固定 `draft -> qualifying -> awaiting_authorization -> pilot -> canary -> promoted`；任一运行态可因安全、gate、新失败类型或人工介入进入 `suspended`，只可经重新qualification/Authorization/canary回归，或进入 `retired`。运行中不动态fallback。
- `AutonomyPolicy` 固定：B0只允许contracts/doctor/prepare；B1允许操作者显式启动单个pilot Batch并自动完成commit链；B2允许scheduler对promoted模板每次drain一个task；B3允许跨多个qualified repo的portfolio调度。升级不扩大Batch禁止边界，必须由静态generation和Authorization激活。
- `OperatorActionCatalog` 至少定义 login、environment requalify、authorization activate、repo requalification、template suspension、reconcile、cleanup、ownership repair、maintenance recovery、activation rollback、cutover/rollback、platform suspend/resume/kill-current；每项固定reason code、scope、前置guard、唯一建议命令、去重key和通知策略，DB不得保存自由文本reason。
- 正常受管Native流程固定为 durable `maintenance_drain_requested` -> 停止新Batch claim -> 等待当前attempt自然terminal -> 取得global capacity -> repo mutex/maintenance begin -> managed Native launcher -> maintenance-end scan -> quick verify或requalification -> release capacity -> resume unaffected scheduling。目标repo失败进入 `repo_requalification_required`，模板失败进入 `template_qualification_suspended`，其他repo继续运行。
- 裸Codex/Git写操作明确unsupported，不安装全局hook；操作者必须使用受管 `native run`，或先durable platform suspend和repo maintenance。Native session由 `OperatorWorkSession` 记录scope、SID hash、generation、reason code、start/end和evidence，但不保存prompt。
- `kill-current` 是独立紧急命令，正常maintenance不得默认调用。副作用前必须durable写入 `OperatorWorkSession`、`OperatorAction`、`FencedActionIntent(terminate_job)`并flush；随后才可终止Job、持续排空、seal journal和handoff recovery。Controller崩溃后新fence只能收养同一个terminate action。
- 公共CLI固定为：`runtime install|status|activate|rollback`；`platform setup|login|logout|auth-status|model-probe|sandbox-probe|qualify|doctor|suspend|resume|kill-current`；`environment register|verify|status|retire`；`repo register|verify|qualify|cutover|rollback|ownership-status|ownership-repair|maintenance enter|status|exit`；`native run`；`authorization activate|revoke|status|verify`；`batch prepare|submit|drain|status|cancel|retry|reconcile|resolve|doctor`；`contracts catalog|verify`；`gates run|verify`；`evidence verify`；`backup create|verify|restore-drill`；`eval baseline|compare`；`schedule install|status|run-now|remove`；只读 `compat verify`。
- 所有CLI提供稳定 `--json`；scheduler强制JSON。退出码固定为0成功/无ready task，2输入或状态契约错误，3 policy/Authorization/ownership/environment/stale/manual-recovery阻断，4 gate/evidence/eval/attempt失败，5 unavailable/needs_auth/qualification_suspended，6 platform_incompatible；非零返回稳定reason code、evidence locator和唯一建议命令。
- Scheduler使用当前用户InteractiveToken、Limited、Hidden、StartWhenAvailable、IgnoreNew，每5分钟最多drain一个task，ExecutionTimeLimit 12600秒；全局capacity为1。Scheduled runtime禁止login/logout、environment preparation、ownership repair、cutover、maintenance和kill-current。
- P0 Truth Reset：先批准规范artifact，再原子同步AGENTS、README、PRD、架构、roadmap、plan、backlog、acceptance、planning status、selector/verifier；queue改为 `MINIMUM-OPERATOR-COMMIT-ONLY-TRANSITION`，selector返回 `implement_legacy_guard_first`，移除旧queue/supplemental token硬编码，并将旧runtime_v2路径降为legacy历史入口。
- P0 Legacy Guard：实现ownership schema/catalog和legacy adapter，覆盖legacy claim、lease、worktree、executor、writer、repo mutation、Git、commit、closeout和cleanup入口；枚举现有repo并注册为legacy owner。Legacy/new交叉conformance、crash、cutover和rollback drill未通过前，新Batch claim硬阻断。
- P1依次实现 Contracts/Storage、Installer/Platform/Execution、Git/Recovery/Operations；每片尽量不超过5个主要文件，但它只是审查目标，不是机械失败门。随后执行Full Q0；P2完成一个低风险self-host commit-only vertical slice；P3完成至少5个scheduled self-host任务。
- P4资格化至少两个真实目标repo并完成30次scheduled observations；P5逐仓CAS cutover。全部目标repo cutover、零legacy active/closing且rollback drill通过后旧DB只读；连续30天零legacy调用后删除旧写面，compat reader、历史evidence和task refs永久只读保留。
- Runbooks必须分别覆盖 needs_auth、needs_environment、platform incompatible/qualification suspended、recovery/reconcile/cleanup、ownership repair、maintenance、emergency kill、activation rollback、backup restore、cutover/rollback；每份包含精确命令、前置guard、stop condition、证据和仅回滚本次动作的入口。

## 5. 门禁、崩溃验证、指标与最终裁决

- 新包固定门禁顺序为：`uv build --offline --project runtime/local-ai-runtime runtime/local-ai-runtime`；`uv run --frozen --offline --project runtime/local-ai-runtime python -m pytest`；`uv run --frozen --offline --project runtime/local-ai-runtime python -m local_ai_runtime contracts verify`；`uv run --frozen --offline --project runtime/local-ai-runtime ruff check runtime/local-ai-runtime`；`uv run --frozen --offline --project runtime/local-ai-runtime pyright --project runtime/local-ai-runtime`；`python scripts/verify-planning-status.py`；`python scripts/select-next-work.py`；`git diff --check`。触及legacy guard时补跑 `uv run --project ./runtime/host-orchestrator python -m pytest`。

- Full Q0在install/activate及Codex/Git/Windows/sandbox/adapter/policy变化时运行；quick preflight每次drain/attempt运行；daily canary在受管临时repo验证allow/deny读写、`trust_level=untrusted`、project config/hooks隔离、provider网络与task断网、keyring refresh、feature/tool inventory、ephemeral disk diff、Job kill、DLL loading、reparse/hardlink、limits、Git对象和deterministic commit。

- Implementation Acceptance要求代码、migrations、CLI、runbooks、legacy conformance、crash tests、backup/restore、全部固定门禁和rollback证据绿色；不允许用repo-side simulation、legacy evidence或probe-only结果替代。完成后仍不得进入P2，直到Full Q0通过。
- 安全硬门为未授权写入、已声明/已检测敏感信息泄漏、重复writer、冲突object/commit/ref、错误cleanup、未记账Git副作用和 `successful_unauthorized_egress` 全部为0。安全失败按GuardCatalog作用域暂停global/repo/template；普通资源或进程失败只影响attempt，禁止把所有失败一律升级为平台不兼容。

- 效率必须使用至少12个相同snapshot、controller prompt、gate、qualification和任务分布的严格配对case，固定seed counterbalance，每个声明模板族至少3个case且不得退化。`net_operator_minutes_per_success = 窗口内提交、监控、auth、qualification、失败/取消/恢复、模板和runtime维护的人工活动分钟总和 / commit-ready成功数`；重叠人工区间只计一次，失败人工计入分子，纯等待和初始研发成本单列。要求下降至少50%，并满足成功任务/操作者小时提高至少50%或P50周期下降至少30%。
