# 社区参考源码策略

## 结论

应该拉本地参考源码，但不应该放在 `D:\CODE\local-ai-dev-orchestrator` 仓里。

推荐位置：

- `D:\CODE\external\hermes-agent-references`

而不是：

- `D:\CODE\local-ai-dev-orchestrator\vendor\...`
- `D:\CODE\local-ai-dev-orchestrator\external\...`

## 为什么不放在维护仓里

`D:\CODE\local-ai-dev-orchestrator` 的职责应当保持单一：

- 保存方案定稿
- 保存实施计划
- 保存任务清单
- 保存本机验收快照
- 保存维护脚本和索引

如果把上游源码也塞进来，会立刻带来这些问题：

- 上游历史与本地维护历史混在一起
- 大量 vendor 更新污染当前仓提交历史
- 以后做 `git status`、差异复盘、回滚和证据归档都会变脏
- 拉官方更新、切 tag、做原始 diff 也更不直观

## 推荐拉取对象

### 1. Core：当前主链默认参考面

这些仓属于当前主链的默认参考面，适合默认保留、默认刷新、默认先看：

- `hermes-agent`
- `codex`
- `modelcontextprotocol`
- `servers`

用途：

- `hermes-agent`：Hermes 官方运行时、Docker、配置和行为真值
- `codex`：主执行层行为真值
- `modelcontextprotocol`：MCP 协议本体真值
- `servers`：官方 MCP server 实现参考

### 2. Secondary：保留，但不属于默认刷新集合

这些仓仍值得保留，但不应继续和 core 仓享受同等默认地位：

- `skills`
- `hermes-agent-self-evolution`

用途：

- `skills`：以后吸收 `skills-drafts/`、沉淀可复用 skill 时对照结构
- `hermes-agent-self-evolution`：未来学习层、自进化与记忆提升方向储备

### 3. Conditional：条件保留

当前只建议条件保留：

- `openclaw`

用途：

- 只有当消息入口、渠道层或社区集成成为真实需求时再升级为活跃参考
- 当前 v1 主链并不依赖它

当前参考入口：

- 官方仓：[OpenClaw GitHub](https://github.com/openclaw)

## 当前不建议并入本参考架的社区项目

以下项目不建议加入 `D:\CODE\external\hermes-agent-references`：

- `AstrBot`
- `NapCat-Docker`
- `NapCatQQ`
- `aiocqhttp`

原因：

- 它们解决的是 QQ / OneBot / IM 渠道层问题，不是当前主链的默认维护面
- 当前仓已经明确把消息入口能力延后处理
- 本机已有独立参考架：`D:\CODE\external\qq-codex-bot-references`

如果以后只是要判断 QQ / NapCat / AstrBot 的实现边界，应直接复用那套专属参考架，而不是在这里重复克隆

## 建议目录

推荐以后在 `D:\CODE\external` 下固定出一个子树：

```text
D:\CODE\external\
  hermes-agent-references\
    hermes-agent\
    codex\
    modelcontextprotocol\
    servers\
    skills\
    hermes-agent-self-evolution\
    openclaw\
```

## 更新纪律

参考仓只做三件事：

1. 保留上游原始 git history
2. 需要时拉最新 tag / release / docs truth
3. 把有价值的结论回写到本维护仓文档里

默认刷新纪律：

- 默认刷新只覆盖 `core`：`hermes-agent`、`codex`、`modelcontextprotocol`、`servers`
- `skills`、`hermes-agent-self-evolution`、`openclaw` 仅在本次任务明确需要时再显式加入刷新范围

不要在参考仓里混入本机长期方案、AgentBridge 快照或私有配置。

## 当前建议

当前最合理的做法是：

- 现在就建立 `D:\CODE\local-ai-dev-orchestrator` 维护仓
- 现在就把本机方案与已验收快照固化进来
- 现在就把高价值上游参考仓拉到 `D:\CODE\external\hermes-agent-references`
- `OpenClaw` 即便已拉取，也继续按“延后能力”对待，不进入 v1 主链
- `NapCat/AstrBot` 这类 QQ 渠道层项目继续留在其专属参考架，不并入本参考架
