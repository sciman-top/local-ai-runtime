# 社区参考源码策略

## 结论

应该拉本地参考源码，但不应该放在 `D:\CODE\hermes-agent` 仓里。

推荐位置：

- `D:\CODE\external\agent-runtime-references\hermes-agent`
- `D:\CODE\external\agent-runtime-references\openclaw`

而不是：

- `D:\CODE\hermes-agent\vendor\...`
- `D:\CODE\hermes-agent\external\...`

## 为什么不放在维护仓里

`D:\CODE\hermes-agent` 的职责应当保持单一：

- 保存方案定稿
- 保存实施计划
- 保存任务清单
- 保存本机验收快照
- 保存维护脚本和索引

如果把上游源码也塞进来，会立刻带来这些问题：

- 上游历史与本地维护历史混在一起
- 大量 vendor 更新污染当前仓提交历史
- 以后做 `git status`、差异复盘、回滚和证据归档都会变脏
- 拉官方更新、切 tag、做原始 diff 也更不直观

## 推荐拉取对象

### 1. 必拉：Hermes 官方仓

用途：

- 对照官方 release、Docker 路径、配置变化、脚本行为与上游真实边界
- 后续升级时做“本机维护层 vs 上游实现层”的 diff

当前参考入口：

- 官方仓：[NousResearch/hermes-agent](https://github.com/NousResearch/hermes-agent)
- 官方文档：[Hermes Agent Docs](https://hermes-agent.nousresearch.com/docs/)

### 2. 条件拉：OpenClaw 官方仓

用途：

- 只有当消息入口、渠道层或社区集成成为真实需求时再拉
- 当前 v1 主链并不依赖它

当前参考入口：

- 官方仓：[OpenClaw GitHub](https://github.com/openclaw)

## 建议目录

推荐以后在 `D:\CODE\external` 下固定出一个子树：

```text
D:\CODE\external\
  agent-runtime-references\
    hermes-agent\
    openclaw\
```

## 更新纪律

参考仓只做三件事：

1. 保留上游原始 git history
2. 需要时拉最新 tag / release / docs truth
3. 把有价值的结论回写到本维护仓文档里

不要在参考仓里混入本机长期方案、AgentBridge 快照或私有配置。

## 当前建议

当前最合理的做法是：

- 现在就建立 `D:\CODE\hermes-agent` 维护仓
- 现在就把本机方案与已验收快照固化进来
- 后续再把 `Hermes` 官方仓拉到 `D:\CODE\external\agent-runtime-references\hermes-agent`
- `OpenClaw` 继续按需，不抢先引入
