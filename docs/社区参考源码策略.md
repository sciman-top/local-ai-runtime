# 社区参考源码策略

## 结论

应该拉本地参考源码，但不应该放在 `D:\CODE\local-ai-dev-orchestrator` 仓里。

推荐位置：

- `D:\CODE\external\local-ai-dev-orchestrator-references`

而不是：

- `D:\CODE\local-ai-dev-orchestrator\vendor\...`
- `D:\CODE\local-ai-dev-orchestrator\external\...`

## 为什么不放在维护仓里

`D:\CODE\local-ai-dev-orchestrator` 的职责应当保持单一：

- 保存方案定稿
- 保存实施计划
- 保存任务清单
- 保存本机验收快照
- 保存维护脚本和索引

如果把上游源码也塞进来，会立刻带来这些问题：

- 上游历史与本地维护历史混在一起
- 大量 vendor 更新污染当前仓提交历史
- 以后做 `git status`、差异复盘、回滚和证据归档都会变脏
- 拉官方更新、切 tag、做原始 diff 也更不直观

## 推荐拉取对象

### 1. Core Mainline：当前主线默认参考面

这些仓属于当前主链的默认参考面，适合默认保留、默认刷新、默认先看：

- `codex`
- `plugins`
- `modelcontextprotocol`
- `servers`

用途：

- `codex`：主执行层行为真值
- `plugins`：当前 Codex plugin / hook / skill-only plugin 打包示例真值
- `modelcontextprotocol`：MCP 协议本体真值
- `servers`：官方 MCP server 实现参考

### 2. Secondary / Compatibility：保留，但不属于默认刷新集合

这些仓仍值得保留，但不应继续和 core 仓享受同等默认地位：

- `openai-agents-python`

用途：

- `openai-agents-python`：多 agent、handoff、guardrail、session、sandbox 设计参考

### 2.5 Governance Sidecar：正式 companion，但不进入默认刷新集合

- `governed-ai-coding-runtime`

用途：

- gate / evidence / selector / preflight 治理机制参考
- 产品主线与治理增强层分层方法参考

当前定位：

- 当前仓正式的 `governance-sidecar` companion
- 通过 manifest `absolute_path` 纳入参考架
- 默认不进入 `default_refresh_set`

### 3. Compatibility：保留为兼容线真值

- `hermes-agent`

用途：

- `hermes-agent`：Hermes/AgentBridge 兼容线运行时、Docker、配置和行为真值

### 4. Archive / On-Demand：保留，但不再是活跃默认面

这些仓可以继续保留，但不建议继续当作当前主线的活跃参考：

- `skills`
- `hermes-agent-self-evolution`
- `openclaw`

用途：

- `skills`：旧 `skills` 结构回看；上游已 deprecated，当前官方主入口已转向 `plugins`
- `hermes-agent-self-evolution`：未来学习层、自进化与记忆提升方向储备
- `openclaw`：只有当消息入口、渠道层或社区集成成为真实需求时再升级为活跃参考

### 5. Conditional Not Cloned：暂不拉取

- `claude-code`

用途：

- 只有当 `claude_glm` review adapter 进入真实编码队列时，再补这类官方参考面

当前参考入口：

- 官方仓：[OpenClaw GitHub](https://github.com/openclaw)

## 当前不建议并入本参考架的社区项目

以下项目不建议加入 `D:\CODE\external\local-ai-dev-orchestrator-references`：

- `AstrBot`
- `NapCat-Docker`
- `NapCatQQ`
- `aiocqhttp`

原因：

- 它们解决的是 QQ / OneBot / IM 渠道层问题，不是当前主链的默认维护面
- 当前仓已经明确把消息入口能力延后处理
- 本机已有独立参考架：`D:\CODE\external\qq-codex-bot-references`

如果以后只是要判断 QQ / NapCat / AstrBot 的实现边界，应直接复用那套专属参考架，而不是在这里重复克隆

## 更新纪律

参考仓只做三件事：

1. 保留上游原始 git history
2. 需要时拉最新 tag / release / docs truth
3. 把有价值的结论回写到本维护仓文档里

默认刷新纪律：

- 默认刷新只覆盖 `core-mainline`：`codex`、`plugins`、`modelcontextprotocol`、`servers`
- `openai-agents-python` 与 `hermes-agent` 仅在本次任务明确涉及 multi-agent / compatibility 时再显式加入
- `governed-ai-coding-runtime` 仅在本次任务明确涉及 gate / evidence / selector / preflight / reference governance 时再显式加入
- `skills`、`hermes-agent-self-evolution`、`openclaw` 仅在本次任务真的需要回看 archive / 渠道储备时再显式加入刷新范围

不要在参考仓里混入本机长期方案、AgentBridge 快照或私有配置。

当前最合理的做法是：

- 现在就建立 `D:\CODE\local-ai-dev-orchestrator` 维护仓
- 现在就把本机方案与已验收快照固化进来

- 现在就把 `governed-ai-coding-runtime` 作为正式 `governance-sidecar` companion 纳入 references manifest
- 现在就把外层参考架从 `hermes-agent-references` 更名为与当前主线一致的名字
- 现在就补 `plugins` 与 `openai-agents-python` 两个更贴近当前主线的官方参考面
- `skills` 继续保留，但降级为 archive/on-demand，不再当作当前官方主入口
- `OpenClaw` 即便已拉取，也继续按“延后能力”对待，不进入 v1 主链
- `NapCat/AstrBot` 这类 QQ 渠道层项目继续留在其专属参考架，不并入本参考架
